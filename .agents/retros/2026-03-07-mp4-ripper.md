# Retrospective: mp4-ripper

__Date:__ 2026-03-07
__Scope:__ Full project — scaffold, download engine, GUI, integration, QA, retro

## Summary

Cross-platform YouTube-to-MP4 desktop app built and shipped in a single session. All three caption modes (none, SRT, burned-in) validated against real YouTube videos on a Windows 11 workstation. Product Owner confirmed the app solved a problem they had been deferring for a year.

## What Went Well

- Stack selection (Python + yt-dlp + Tkinter) was correct on the first pass — no backtracking
- Review-driven planning (TASK_LIST.md before code) confirmed the tri-state caption design before implementation
- Real-world testing caught three bugs no synthetic test would have found
- Product Owner had a working 1080p download with audio in the first run
- VS Code extension environment on Windows 11 worked flawlessly; PowerShell guidance was correct

## What Could Be Improved

- Initial window height too short (720px) — Download button clipped on first launch
- yt-dlp SRT embed behavior assumed wrong on first implementation — required architecture change
- First `_clean_srt` pass only handled prefix rolling-word pattern; same-timestamp stacking required a second revision
- README should explicitly call out that `yt-dlp` CLI requires `python -m yt_dlp` when installed via pip
- README caption table should note VLC as the recommended player for SRT soft subtitles on Windows

## Learnings Extracted

- L1: yt-dlp silently embeds subtitles during merge — requires separate subtitle fetch pass
- L2: YouTube auto-captions need a three-pass SRT cleaner (prefix drop, same-timestamp merge, overlap trim)
- L3: FFmpeg `subtitles=` filter breaks on Windows drive-letter colons — use `cwd=` workaround
- L4: Startup dependency checks beat runtime error handling for external CLI tools
- L5: Windows 11 Media Player v11 is not WMP 12 — VLC is the correct SRT reference player
- L6: Tkinter requires `after(0, ...)` for all background thread GUI updates
